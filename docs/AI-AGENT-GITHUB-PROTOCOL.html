§03 Agent Workflow

The Only Steps You Are Authorized to Take

  1. Receive the task
    User tells you exactly what to change — a file rename, a text update, a new file to add. Nothing else. If it's ambiguous, ask one clear question before proceeding.
  2. Clone the repo using the token
    Clone fresh every time. Never assume your local copy is up to date. Confirm the repo URL matches github.com/aiandi49/[repo-name] before running any command.
  3. Make only the requested change
    Touch only the file(s) explicitly mentioned. Do not refactor, rename, reformat, or "improve" anything else. If you see something broken unrelated to the task — report it, don't fix it without permission.
  4. Verify before committing
    Run a quick check to confirm the change looks correct. For text replacements, grep for the old string to confirm it's gone. For new files, confirm they exist in the right folder.
  5. Commit with a clear message
    Write a commit message that says exactly what changed. Example: "Rename CAM to KAM in README and workflow HTML files". Never write vague messages like "update" or "fix".
  6. Push and report back
    Push to main. Then tell the user: what you changed, what files were affected, and what you deliberately left alone. Done. Wait for next instruction.
§04 Rules of Engagement

Do This. Never Do That.

✓
Confirm the repo URL
Always verify you're pushing to github.com/aiandi49/[correct-repo] before running git push.
✗
Never guess the URL
If the repo name isn't 100% confirmed, ask. A wrong push can corrupt the repo and take hours to fix.
✓
Change only what's asked
Scope creep breaks things. If asked to rename a word, rename only that word. Nothing else.
✗
Never touch backend configs
Railway, Render, Supabase, .env files, API keys — these are off limits unless explicitly authorized.
✓
Report what you left alone
After every push, tell the user what you intentionally did NOT change and why.
✗
Never force push without warning
git push --force rewrites history. Only use it if explicitly told to, and warn the user first.
✓
Set git identity every time
Always run git config user.email and user.name before committing. Use aiandi49@github.com.
✗
Never commit .env or tokens
Access tokens, API keys, and .env files must never be committed to the repo. Check .gitignore.
§05 Scope Boundaries

What This Token Does NOT Authorize

🚫 Off Limits — Frontend/Backend Integrations

Do not modify any deployment configs, Railway settings, Render services, Supabase schemas, or environment variables. These are handled separately. If a file change requires a backend restart or redeployment, stop and tell the user.

🚫 Off Limits — Other Repos Without Confirmation

Even though one token covers all repos under aiandi49, do not touch any repo other than the one you were specifically directed to. Always confirm the repo name before cloning.

✅ Authorized — File-Level Changes Only

Renaming text, updating HTML content, adding new files, creating folders, fixing typos, updating README files, adding demo pages. These are all safe and authorized without extra confirmation.
